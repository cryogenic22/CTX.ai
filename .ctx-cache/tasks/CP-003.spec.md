## Smallest test that proves it works

```python
def test_extract_app_finds_two_functions():
    from ctxpack.core.code.parser import parse_python
    from ctxpack.core.code.symbols import extract_symbols, Kind
    result = parse_python(FIXTURE_DIR / "app.py")
    syms = extract_symbols(result)
    funcs = [s for s in syms if s.kind == Kind.FUNCTION]
    assert {s.name for s in funcs} == {"read_user", "create_user"}
```

## Design

```python
class Kind(str, Enum):
    FUNCTION = "function"
    CLASS = "class"
    # methods, properties, etc. added in CP-004

@dataclass(frozen=True)
class Symbol:
    name: str
    kind: Kind
    line_start: int     # 1-indexed (CP-002 convention)
    line_end: int       # 1-indexed inclusive
    byte_start: int     # for source[byte_start:byte_end] slicing
    byte_end: int

def extract_symbols(result: ParseResult) -> list[Symbol]:
    """Top-level functions and classes only. Returns in file order
    (top-to-bottom by byte_start)."""
```

Decisions:

- **`Kind` as `str, Enum`.** Same convention as `ContextLayer` in
  `ctxpack.core.layers` — JSON-serialisable, ordered for stability.
  Adding new kinds later (METHOD, PROPERTY, COMPONENT, HOOK) is
  additive.
- **Byte offsets included now, not later.** Every downstream task
  that needs source text (CP-004 method bodies, CP-005 decorators,
  CP-010 IREntity emitter) will need them. Adding them in CP-003 is
  free; bolting them on later means re-running every test.
- **Top-level only.** CP-003 walks the immediate children of the
  module node. Tree-sitter Python's module node has
  `function_definition` and `class_definition` as direct children
  for top-level defs. Nested functions and class methods are
  CP-004's job; the walker explicitly does NOT recurse.
- **Async functions count as functions.** `async def foo()` parses
  as `function_definition` with an async modifier in tree-sitter
  Python. Same kind for v0 — agents care about "this is a function
  named foo" more than they care about sync-vs-async at the catalog
  level. The signature field (CP-005+) will preserve `async`.
- **Decorated functions count as functions.** Tree-sitter wraps
  decorated defs in a `decorated_definition` node containing the
  decorator(s) + the underlying function/class definition. The
  walker descends through `decorated_definition` to find the
  underlying def; decorators are captured at CP-005.
- **Order: file order (byte_start ascending).** Matches how the
  catalog will render; matches how humans read code.

## Failure modes to red-team in step [5]

- Decorated function would silently disappear if the walker doesn't
  descend through `decorated_definition`.
- Async function would silently disappear if the walker only matches
  the plain `function_definition` type name string.
- Multiple classes in one file produce duplicate-named symbols if
  someone refactors badly — at v0, this is "same name twice" not a
  crash; CP-009 will disambiguate.
- A class with a method should produce ONE symbol (the class) at
  CP-003. The method should NOT appear (that's CP-004).
- Empty file → empty list, not a crash.
- File with only comments → empty list.
- File with only a top-level assignment (`x = 1`) → empty list at
  CP-003 (top-level constants are not in v0 scope).
- Source-byte round-trip: `source[byte_start:byte_end]` for a
  function symbol must produce the function's source text.
- Byte and line ranges must agree (line_start matches the line
  containing byte_start, etc.).

**Red-team adds** (already in step [3]):
- Methods NOT extracted at top level (one symbol for `class Foo: def bar`).
- Nested functions NOT extracted.
- Decorated classes extracted (walker descends through decorated_definition).
- Double-decorator chains do NOT lose the underlying function.

**Notes for downstream**:
- CP-004 (methods) walks `class_definition.body` children, same
  unwrap-via-`decorated_definition` pattern.
- CP-005 (decorators) reads the decorator children that we deliberately
  ignored at CP-003; the function symbol's byte range is body-only,
  so decorator capture has its own byte range that lives on the
  IREntity decorator field.
- CP-009 will add `<file>::<dotted>` disambiguation. Until then,
  `Symbol.name` is the local identifier only.
- `Kind` enum is additive (`str, Enum`, JSON-serialisable). New
  values land in CP-004/008/012 without breaking existing
  callers.
