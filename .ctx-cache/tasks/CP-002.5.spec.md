## Tokeniser choice — decision summary

**Pinned: `tiktoken` with encoding `cl100k_base`.**

Rationale (full version in the ADR):

1. **Already de facto in this repo.** `ctxpack/benchmarks/metrics/cost.py`
   uses cl100k_base via tiktoken for BPE counts. The v3 whitepaper's
   "BPE tokens" metric is a cl100k count. Using anything else creates
   measurement drift between the code packer and the existing eval.
2. **Offline.** No API call, no rate-limit failure mode.
3. **Cross-model proxy.** Claude tokens correlate with cl100k counts
   within ~10–15% in practice. Imperfect, but the discipline is
   "single source of truth" not "perfect alignment with Claude's bills."
4. **Stable.** cl100k_base is frozen since GPT-4 launched; future
   `tiktoken` upgrades won't shift counts under us.

**Risk**: cl100k drift vs. Claude's actual tokeniser will surface as
budget over- or under-shoot when the agent actually consumes the
served pack. Acceptable for v0; flagged in the ADR.

**Override hook**: `count_bpe(s, *, encoding=...)` accepts an
optional encoding name so power users can swap without forking, but
the default and every internal caller stays on cl100k_base.

## Smallest test that proves it works

```python
def test_count_bpe_known_reference_strings():
    from ctxpack.core.code.tokens import count_bpe
    # Reference counts computed once via cl100k_base, frozen here so
    # a future tiktoken upgrade that shifted counts would be caught.
    assert count_bpe("hello") == 1
    assert count_bpe("") == 0
    assert count_bpe("def foo(x: int) -> str: return str(x)") == 13
```

## Design

```python
# ctxpack/core/code/tokens.py

_ENCODER_NAME = "cl100k_base"
_ENCODER = None  # lazily initialised, module-singleton

def count_bpe(s: str, *, encoding: str | None = None) -> int:
    """Return the BPE token count for s under the pinned encoding.

    Single source of truth for every BPE measurement and budget
    enforcement in the code packer. Internal callers MUST use this
    helper; bypassing it creates measurement drift.
    """
```

Decisions:

- **Module-level encoder singleton, lazily initialised.** Same
  pattern as the tree-sitter parser singleton (CP-002): cheap on
  first call, reused for every subsequent call, no per-call setup
  cost. Thread-safety follows tiktoken's documented contract
  (encoders are safe to share across threads).
- **Empty string returns 0.** Saves every caller a length check.
  tiktoken's encode("") returns []; len([]) == 0 — but pinning it
  in a test guards against a future change.
- **`encoding` kwarg.** Optional override so a caller can compare
  cl100k vs o200k without forking the helper. Default stays
  cl100k_base; every CTX-internal call site omits the kwarg.
- **Reference strings.** Five chosen to span: pure ASCII, empty,
  whitespace-heavy code with operators, non-ASCII identifiers
  (utf-8 round-trip), long type-union signature (the catalog soft-cap
  case). Pinning these protects against silent count drift in
  future tiktoken upgrades.
- **`tiktoken` becomes a hard dep for the `code` extra.** Already
  conventional in this repo; no determinism risk because
  cl100k_base is frozen.

## Failure modes to red-team in step [5]

- Caller passes bytes instead of str → tiktoken raises; we surface a
  clear TypeError rather than letting tiktoken's error bubble.
- Caller passes non-string (int, None) → TypeError, not a confusing
  attribute error.
- Threaded callers race on first-time encoder init → singleton must
  be safe (tiktoken loads its BPE table once and that table is
  read-only thereafter).
- `count_bpe` called repeatedly on the same string returns the same
  number (determinism).

**Red-team additions** (step [5]):
- Invalid encoding name raises rather than silently falling back —
  catches typos in researcher A/B configs.
- Whitespace-only strings count as ≥1 token (not 0) — pins the
  edge case where someone might assume empty-ish-looking input is 0.
- Architectural pin via test: no `import tiktoken` allowed elsewhere
  in `ctxpack/core/code/`. Allowlist documented in the test
  docstring (the helper itself; benchmarks/metrics/cost.py and
  benchmarks/tokenizer_mapping.py are sibling tools by design).
