# CP-005 — Decorator capture on functions and methods

**Status**: shipped
**Backlog entry**: CP-005 (Week 1, deps CP-004)
**Effort**: 0.5d

## Acceptance
- `@app.get("/foo")` produces `decorators=["app.get"]` with `decorator_args=[{"path":"/foo"}]`.

## Design
Extend `Symbol` with `decorators: tuple[Decorator, ...] = ()`. `Decorator` is a frozen dataclass with `name` (dotted), `args` (positional literals as repr strings), `kwargs` (tuple of `(key, repr)` pairs), `line` (1-indexed).

Walker change: when we hit `decorated_definition`, extract every `decorator` child before unwrapping to the function/class, and attach the captured list to the resulting Symbol.

For methods inside a class body, same pattern: walk class-body children, decorated_definition children get their decorators captured and attached to the inner method.

## Non-goals
- Interpreting decorator semantics (CP-006 reads `app.get` and emits route metadata; CP-007 reads `Depends`; CP-008 looks at base classes for Pydantic). CP-005 just produces structured decorator data.
- Decorator factory calls with computed args (e.g. `@some_func(x + 1)`). We capture the literal source repr; downstream consumers decide whether to evaluate.

## Tests in scope
- Plain decorated function captures `["app.get"]` + kwargs `("path", '"/foo"')`.
- Multiple decorators preserved in source order.
- `@property` / `@staticmethod` (no args) → name only, empty args/kwargs.
- Method-on-class decorators flow through.
- Undecorated function: `decorators == ()`.
- Decorator with positional + keyword args.
