# CP-004 — Class method + class-attribute extraction

**Status**: shipped
**Backlog entry**: CP-004 (Milestone v0, Week 1)
**Deps**: CP-003 (Symbol/Kind, top-level walker)
**Effort**: 0.5d

---

## Acceptance (from backlog)

- `Foo.bar` symbols emitted with `kind=method`.
- Class-level attrs captured as `kind=class_attribute`.
- Including `__init__`, properties, staticmethods (i.e. the walker
  doesn't skip dunders or decorated methods).

## Non-goals

- Decorator capture / classification (regular vs property vs static).
  All methods get `kind=METHOD` at CP-004. CP-005 inspects decorators
  separately as metadata on the IREntity.
- Inheritance / MRO. We only see what's syntactically inside the class
  body; we don't trace `class Bar(Foo)` to inherit Foo's methods.
- Inner classes inside classes. Out of v0 scope.
- Pydantic `BaseModel` field detection — that's CP-008, which uses
  the CLASS_ATTRIBUTE entries CP-004 produces as input.

## Smallest test that proves it works

```python
def test_widget_method_and_attribute_extraction():
    from ctxpack.core.code.parser import parse_python
    from ctxpack.core.code.symbols import extract_symbols, Kind
    syms = extract_symbols(parse_python(FIX / "widget.py"))
    names = {(s.kind, s.name) for s in syms}
    assert (Kind.CLASS, "Widget") in names
    assert (Kind.METHOD, "Widget.__init__") in names
    assert (Kind.METHOD, "Widget.tick") in names
    assert (Kind.CLASS_ATTRIBUTE, "Widget.MAX_TICKS") in names
```

## Design

```python
# Kind additions (str, Enum stays JSON-stable):
Kind.METHOD          = "method"
Kind.CLASS_ATTRIBUTE = "class_attribute"

# extract_symbols flow:
for child in module.children:
    target = _unwrap(child)
    if target is None:
        continue
    sym = _make_symbol(target)
    if sym is None:
        continue
    out.append(sym)
    if target.type == "class_definition":
        out.extend(_class_body_symbols(target, class_name=sym.name))

# _class_body_symbols(class_node, class_name) -> list[Symbol]:
#   body = class_node.child_by_field_name("body")
#   for grandchild in body.children:
#     unwrap decorated_definition
#     function_definition -> METHOD, name = f"{class_name}.{fn_name}"
#     expression_statement containing typed_default_parameter or
#       assignment with annotation -> CLASS_ATTRIBUTE,
#       name = f"{class_name}.{attr_name}"
```

Decisions:

- **Method naming `Class.method`**, not `method` alone. The dot is the
  convention CP-009 will extend to `file::Class.method`. Choosing the
  dot now (rather than `::method`) keeps the eventual final form
  natural to read.
- **Walk class body via the `body` field**, not via iterating raw
  children. The `body` is a `block` node and its children are the
  things we care about — direct children of the class node include
  things like keyword arguments to the class definition itself (e.g.
  `class Foo(Bar, metaclass=Meta):` has those as siblings of the body
  block).
- **Class attribute detection** uses tree-sitter's child-type
  matching: an `expression_statement` whose first child is either
  `assignment` (e.g. `X = 1`) or contains a typed
  identifier (e.g. `id: int`). Both shapes appear in real code;
  Pydantic favours `id: int`, dataclasses favour `id: int = field(...)`.
- **Walking does not recurse into nested classes or nested functions.**
  Methods INSIDE a method are out of scope. Same posture as CP-003.

## Failure modes red-teamed in [5]

- A method with decorators (`@property`, `@staticmethod`, `@classmethod`)
  must be extracted — walker descends through `decorated_definition`
  on the class-body level too. Decorator type doesn't affect the Kind
  (all METHOD).
- A method named `__init__` (or any dunder) must be extracted.
- An async method must be extracted (`async def` is still
  `function_definition`).
- `class Foo(Bar):` parent class clause must not be treated as
  something extractable.
- Naked module-level expression in class body (e.g. a docstring
  string literal) must NOT be emitted as CLASS_ATTRIBUTE.
- A method whose name collides with a method in another class
  produces two distinct symbols (`Foo.tick` vs `Bar.tick`), no
  dedup.
- Properties / setters / deleters with same name (`@x.setter` after
  `@property`) — both get extracted, each as METHOD. CP-005 will
  distinguish them via decorator metadata.
