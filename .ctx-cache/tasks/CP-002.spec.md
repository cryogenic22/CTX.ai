## Acceptance (from backlog)

- `parse_python(path) → tree-sitter.Tree`.
- Handles syntax errors gracefully (returns partial tree + warnings list).

## Non-goals

- No symbol extraction (CP-003).
- No decorator/route/FastAPI awareness (CP-005–CP-007).
- No call graph (CP-014).
- Not a generic multi-language wrapper — Python only. CP-011 lands TSX.
- No incremental parsing. The wrapper parses a file once, end-to-end.
  Tree-sitter supports incremental reparse for editor use cases, but
  the packer's edit-detection is file-level via IncrementalPacker
  (SHA-256 + mtime), so per-edit incremental parsing earns nothing.

## Acceptance deviation worth flagging

The backlog row says `parse_python(path) → tree-sitter.Tree`. The
implementation returns a `ParseResult` dataclass wrapping the tree
because downstream extractors need the source bytes (for offset →
substring) and the warnings list alongside the tree, and threading
those through three return values is worse than a 3-field dataclass.
`ParseResult.tree` is the literal tree-sitter Tree, so the contract
is met in spirit; the wrapper just delivers more.

## Smallest test that proves it works

```python
def test_parse_python_returns_module_root():
    from ctxpack.core.code.parser import parse_python
    result = parse_python(FIXTURE / "app.py")
    assert result.tree.root_node.type == "module"
    assert not result.tree.root_node.has_error
    assert result.warnings == []
```

## Design

```python
@dataclass(frozen=True)
class ParseWarning:
    message: str        # human-readable
    line_start: int     # 1-indexed for human consumption
    line_end: int       # 1-indexed inclusive
    column_start: int   # 0-indexed (matches tree-sitter convention)
    column_end: int

@dataclass
class ParseResult:
    path: Path
    tree: "tree_sitter.Tree"
    source: bytes       # the bytes the parser consumed (utf-8)
    warnings: list[ParseWarning]
    @property
    def root(self): return self.tree.root_node

def parse_python(path: str | Path) -> ParseResult:
    ...
```

Decisions:

- **Module-level parser singleton.** Tree-sitter `Parser` objects are
  cheap to reuse and not thread-safe per docs. CP-002 keeps it simple
  with a module-global `_PY_PARSER` lazy-initialised on first call,
  guarded by a lock. Thread-safety is a "fix it when it bites" item.
- **utf-8 decoding boundary.** Files are read as bytes and passed
  straight to tree-sitter. Source bytes are stored on the result so
  callers (CP-003+) can extract `node.start_byte:node.end_byte`
  substrings without re-reading the file or re-encoding.
- **Warning generation.** Walk the tree once; any node where
  `is_error` or `is_missing` is True emits a warning. The walker is
  iterative (stack-based) to avoid blowing the call stack on
  large files.
- **Line indexing.** Tree-sitter `Point.row` is 0-indexed; humans
  expect 1-indexed line numbers. `ParseWarning.line_start` /
  `line_end` are 1-indexed; column stays 0-indexed since editors
  vary on column convention but agree on lines.
- **Optional dependency.** `tree-sitter` and `tree-sitter-python`
  go into `[project.optional-dependencies].code`. The parser module
  imports them at top-level (not lazy) so import-time failure is
  fast and obvious. The core package importability test from CP-001
  is preserved because it doesn't import `parser` — only
  `ctxpack.core.code` (the empty `__init__.py`).

## Failure modes to red-team in step [5]

- File doesn't exist → should raise FileNotFoundError, not bubble a
  tree-sitter binding error.
- File is a directory → IsADirectoryError, not silent.
- Empty file (0 bytes) → returns a tree with `module` root and zero
  children, no warnings.
- File with only comments / only whitespace → same as empty for our
  purposes — must succeed, no warnings.
- File with non-utf-8 bytes (e.g. legacy latin-1) → must surface a
  clear error, not crash inside the parser binding.
- Non-ASCII identifiers → utf-8 should handle these natively;
  ParseResult.source preserves the bytes verbatim.
- File with a syntax error → returns tree with `has_error=True` and
  at least one warning; tree is still usable (partial parse).
- Two consecutive parses of the same file return equivalent results
  (no parser-state leakage between calls). This is the determinism
  invariant the §8.6 CI gate will eventually depend on.

**Red-team additions** (step [5]):
- Nested syntax errors must surface (catches a polarity bug in the
  optimised tree walk where descending only when `has_error` is True
  could skip errors at lower levels).
- Interleaved parses of different files don't cross-pollute parser
  state (singleton parser pattern verified).
- ParseWarning is frozen (sets/dict-key safety for downstream code).
- ParseResult.warnings list is per-instance (mutable-default-arg
  footgun pinned).

**Notes for downstream tasks**:
- `tree-sitter`, `tree-sitter-python` are now optional deps under
  `[project.optional-dependencies].code`. The core package remains
  zero-dep; install via `pip install ctxpack[code]` to use the
  packer.
- `ctxpack.core.code.parser` is the ONLY place tree-sitter is

  to any line/column we surface downstream.
